# Usage and cost

Relay durably owns execution, model call and attempt, usage, retry, compaction, child, wait, approval, and tool lifecycle facts. Rika does not copy those facts into an analytics schema or expose a separate analytics query surface; its local transcript and terminal totals are disposable product projections over Relay replay and inspection.

Rika derives model cost from Relay `model.usage.reported` usage and `model.attempt.completed` provider-cost observations, then exposes totals per Turn, Thread, and terminal session. A provider-reported USD amount is authoritative once for its model attempt. Otherwise Rika estimates cost from the bundled models.dev snapshot, using the reported provider, model snapshot, service tier, context tier, uncached input, cache reads, cache writes when known, and output. Rika does not maintain its own model prices or fetch live pricing while replaying durable events. Unknown models, service tiers, prices, currencies, and malformed token buckets remain unpriced instead of falling back to a generic rate. Child Run usage is attributed to its parent Turn and Thread by traversing Relay's durable child tree.

Token-derived cost values are estimates. Missing total input, uncached input, cache-read, cache-write, or output usage is unknown, not zero. Each source event is counted once by execution-scoped event identity, including on replay, recovery, and duplicate or out-of-order delivery. Thread totals cover all of that Thread's Turns. The terminal's global total covers Turns found in its bounded load of up to one hundred Threads. Rika withholds a global total when that bounded replay is incomplete. The selected Thread is collected independently across all of its Turns, so unrelated global gaps do not hide its cost.

The top-right usage control remains visible while selected Thread usage is loading or unavailable. Clicking it cycles locally through cost, exact total tokens, and active time. Total tokens are input plus output; cache buckets partition input and reasoning already belongs to output. Cost, token, and active-time availability are independent.

Active time is the wall-clock union of working intervals across the selected Thread's complete Relay execution tree. An execution starts contributing on `execution.started` and stops on a durable wait, completion, failure, or cancellation. A later `execution.started` resumes it. Parent and child intervals overlap rather than add, so parallel child Runs do not multiply elapsed time; child work still keeps the timer advancing while its parent waits. Rika reconstructs the value from Relay replay whenever the Thread is opened instead of storing a second timer counter. Missing or malformed lifecycle history makes the timer unavailable rather than estimated.

Thread deletion removes Rika's local product state but does not purge Relay's canonical execution records.
