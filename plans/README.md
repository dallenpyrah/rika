# Implementation Plans

Generated by the improve skill on 2026-07-18, audited at commit `ea247c4`. Each plan is self-contained for an executor with zero prior context: read the plan fully before starting, honor its STOP conditions, run every verification command, and update your row below when done.

These ten plans came out of a deep nine-category audit of the whole repo. The audit found the codebase in genuinely good shape — clean package layering, a disciplined doc system, a strong test suite (the transcript reducer alone has 18 tests), and solid defenses on the scary surfaces (path traversal, command injection, the resident daemon, `apiKeyEnv` redaction). The planned work is the exceptions to that: one real security gap (the workspace config file is untrusted input that isn't gated), a handful of correctness bugs, and three "one seam away" capabilities the architecture already anticipates.

## Read this first: the workspace-trust decision gates the security plans

Plans 001, 002, and 010 all defend the same boundary — an untrusted `.rika/settings.json` in a cloned repo. Plan 010's spike raises the load-bearing question: implement them as **one** first-use "trust this workspace?" gate, or three separate mechanisms? **Decide that (plan 010) before implementing 001 and 002**, so you don't build three parallel gates and then merge them. 001 (permission tightening) is safe and independently shippable today; 002 and the SEC-01 credential-redirection fix should follow the trust-model decision.

## Findings identified but not planned this round

Real findings from the audit, deliberately not turned into plans this round (recorded so they are not re-audited; most are "worth doing later", not "rejected"):

- **God-object splits** — `operation.ts` (2,943-line `productLayer`), `main.ts` (3,248), `execution-backend.ts` (2,268), `adapter.ts` `Surface` (1,464): real, but L refactors that want scene-test safety nets first; do after the bug fixes settle.
- **Transcript reducer O(L²) per-delta work** (`transcript/index.ts`): re-parses the whole growing tool input each delta; real hot path, medium impact for a local single-user app. Defer.
- **Other performance**: per-step full transcript delete-reinsert (`operation.ts:1606`), thread-list full-scan+decode (`thread-repository.ts:213`), unindexed checkpoint `SUM` per page (`transcript-repository.ts:392`). All real, medium; batch into a perf pass later.
- **Model mispricing** (`transcript/index.ts:339`) — `claude`/`fable`/`opus` all priced at `[5,25]`, so Sonnet bills at the Opus rate on the no-explicit-cost fallback path; pair the fix with a default-branch pricing test. Small; not selected this round.
- **Memory/SQL keyset collation split** (`turn-repository.ts:379` filters with `<`, sorts with `localeCompare`) + a real-SQLite page test: latent (ULID-like ids usually agree); worth closing with the test.
- **`mergeSessionPrompt`** (`execution-backend.ts:269`) — O(history²) `JSON.stringify` overlap that can also drop a legitimately repeated message; anchor on message ids.
- **Dependency hygiene**: pin the floating `@ff-labs/fff-node ^0.9.6` into the catalog and make `check-dependencies.ts` read workspace manifests (it only reads root today); delete the empty `packages/diagnostics` and the unused `@effect/sql-mysql2` pin.
- **Wire-boundary typing** (`execution-backend.ts`, 15 `any` casts at the Baton/Relay ACL) + dedupe the 5 copies of the `execution:`-strip helper and the duplicated route helpers; `resident-wire.ts` shows the Schema-decode pattern to copy.
- **README first-run gap**: default operation needs `OPENAI_API_KEY`, which the README never names (it only shows a custom `apiKeyEnv` example).
- **Latent security (defense-in-depth)**: MCP OAuth token file lacks the resident token's symlink/uid hardening (`mcp-oauth.ts:125`); MCP `headers`/`env` secrets stored plaintext with no env indirection; `read_web_page` has no internal-range deny-list (not reachable today — fetch is delegated to a cloud API — but the guard should travel with the tool); resident handshake nonce not tracked for replay. **SEC-04**: plugin/MCP trust stores are in-memory only and `McpRuntime.connect` has no internal trust check — harden _before_ plan 008 wires these paths into execution.
- **Effect v4 beta compatibility matrix** doc (the lockstep `beta.93` + Baton/Relay/OpenTUI pins) — overlaps the already-tracked rika #124.
- **Other direction options** not selected: concurrent read-only thread observer (a genuine product decision, the resident already multiplexes connections); headless workflow runs with structured completion (natural follow-on to plan 009).

## What was audited and found solid (no action)

Recorded so these aren't re-audited: package layering is acyclic (TUI never imports Relay/Baton/persistence); path traversal is well-defended (`resolveContained`/`realPath` containment in the tools); command injection is argv-safe (no `sh -c` interpolation); the resident daemon is loopback-only with a 256-bit token and symlink/uid/mode/inode checks; `apiKeyEnv` keeps only the env-var name and values stay `Redacted`; no secrets in committed files; no prompt-injection content in the repo; the doc system matches the code closely; and `bun run check` is a real, comprehensive gate.
